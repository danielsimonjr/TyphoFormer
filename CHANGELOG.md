Notable changes to this repository, per [Keep a Changelog](https://keepachangelog.com/): new work goes under **[Unreleased]** as it lands, using the categories **Added / Changed / Fixed / Docs**. The project has no versioned releases yet, so the history below is grouped by date, newest first.

## [Unreleased]

### Added
- **IBTrACS global best-track ingestion — the model reaches the West Pacific** (roadmap Item 7, the first data win). `tools/ibtracs_to_csv.py` converts any IBTrACS basin file into the repo's CSV schema, handling IBTrACS's mixed 3/6-hourly cadence (keeps only synoptic 00/06/12/18 UTC steps so the 6-hourly window stays uniform) and spur sub-tracks (`--track=main`); `tools/fetch_ibtracs.sh` downloads + converts a basin in one command. Committed `IBTRACS_WP.csv` (**1,446 West Pacific storms, 52,735 synoptic records since 1980 — 1.75× the 826-storm HURDAT set**, and every one an actual typhoon; pinned `-text` in `.gitattributes`). Measured: the recipe (`--direct`) beats split-matched CLIPER on WPac by **−29.9 km / −11.4% over 6–48h, 5/5 seeds** — a *wider* relative margin than the Atlantic's −7.3% (§22), because West Pacific storms recurve more and dead reckoning fails harder (48h CLIPER 551 vs 483 km). The model helps most on the basin it is named for (FINDINGS §24).
- **`--direct` — direct multi-horizon decoder head** (roadmap Item 5, the biggest accuracy win). Predicts all `pred_len` corrections in one shot from the pooled context (no autoregressive rollout), each anchored at seed constant-velocity `p0+(s+1)·v0` (fc2 zero-init → untrained emits pure seed-CV). Avoids the AR head's rollout error compounding. Gradient-checked (`direct` / `direct+multistep` FD-pass), off by default and golden-neutral when off, records `TF_MODE_DIRECT` (eval self-configures), runs serial and `--threads=N`. Measured 5-seed vs the recipe's `--cv` head: **better at 6h (32.1 vs 35.2, ties CLIPER; σ 0.86 vs 2.71) AND at 48h (217.8 vs 227.9 over 6–48h, −10 km, 5/5 seeds, ~4.8σ)**, worse on no seed, and faster. Erases the model's "loses at 6h" caveat (§14–15) and widens the 6–48h margin over CLIPER from 7.1 to 17.2 km. Excludes `--cv`/`--gru`/`--xattn`. A strong candidate to replace `--cv` as the default decoder (FINDINGS §22).
- **`--swa` / `--swa_start=E` — Stochastic Weight Averaging** (roadmap Item 3, the first accuracy win). Averages the weights over the late-epoch tail (default: second half) instead of keeping the single best-val checkpoint, targeting the flat-val-landscape + chaotic-training regime (FINDINGS §15). LayerNorm (no BatchNorm) means the average needs no stat recompute; early stopping is disabled under `--swa` so the window is defined. Each run reports **both** the best-checkpoint and SWA held-out numbers (a clean per-run A/B) and saves the average as `<save>.swa`. Off by default and bit-neutral when off (golden unchanged). Measured on the recipe across 5 seeds: **neutral at 6h (+1.2 km, 2/5) but −6.0 km at 48h (4/5 seeds)** — the 6–48h mean improves 227.0 → 221.0, nearly doubling the recipe's margin over split-matched CLIPER (7.1 → 14.0 km). Recommended opt-in for long-horizon forecasting (FINDINGS §20).
- **`make CUDA=1 test-cuda` — an *executing* gate for the CUDA backend.** CI can only compile the `.cu` (GitHub runners have no NVIDIA GPU), which is precisely how a backend that could never run survived in the tree. The new target mirrors `test-opencl`: kernel cross-check, then the tensor and full-model gradient checks run **through** the CUDA kernels. `CUDA=1` wires the backend into the top-level Makefile the same way `OPENCL=1` does (`CUDA_ARCH` defaults to `sm_61`; note **CUDA 13.0 dropped Maxwell/Pascal/Volta**, so `sm_61` needs a 12.x toolkit).
- **The backend cross-check is now shared.** `backends/opencl/test_opencl.c` only ever spoke the `tensor.h` seam — nothing in it was OpenCL-specific — so it moved to `backends/test_backend.c` and both `test-opencl` and `test-cuda` build it. One check, whichever backend is linked in.
- **`HURDAT_full.csv` — the real dataset, committed. The "98 storms" ceiling is gone.** `tools/fetch_hurdat2.sh` was run against live NOAA (reachable from Daniel's machine, unlike the dev container that could only 403) and its output is pinned here: **24,177 records across 826 storms, 2004-05-22 → 2025-11-01**, three basins (AL/EP/CP) — versus the bundled `HURDAT_2new_3000.csv` sample's 3,004 records / **98 storms** / 2020-2024. That is **8.4× the storms** and includes the full 2025 season.
  - Sources pinned verbatim as `hurdat2_atl.txt` / `hurdat2_epac.txt` (NOAA's `hurdat2-1851-2025-02272026.txt` and `hurdat2-nepac-1949-2025-02272026.txt`, the 2026-02-27 release). NOAA **revises** HURDAT2 between releases, so committing the raw bytes is what makes a future re-run reproducible rather than merely repeatable — re-running the fetch script later can legitimately yield different data.
  - Integrity checked, not assumed: 11,885 Atlantic + 12,292 Pacific = 24,177 rows, exactly one header, both basins present. The only blank column is `record_identifier` (97.9% of rows) — that is normal HURDAT2, a sparse landfall/peak flag; lat, lon, wind, pressure and all wind-radii columns are fully populated.
- `tools/fetch_hurdat2.sh`: one-command dataset scaling — discovers the newest Atlantic and NE Pacific HURDAT2 files from NOAA's directory listing, downloads both, converts via `hurdat2_to_csv.py`, and merges into `HURDAT_full.csv` ready for text-free recipe training. **Now verified end-to-end against live NOAA** (previously only against synthetic two-basin data, because the dev environment's network policy 403s NOAA on every pathway); the newest-file discovery from the directory listing resolved the date-suffixed filenames correctly on the first real run.
- `--tf=E` teacher forcing for the cv rollout: a training step's output state is replaced by the ground-truth position (and truth-implied velocity) with a probability annealed 1→0 over E epochs; forced steps cut the recurrence gradient exactly (truth is constant). Deterministically gradient-checked at force-probability 1.
- `--no_lon` ablation: zeroes `--motion`'s absolute-longitude column (keeps the checkpoint layout) to test whether absolute longitude is climatology signal or a memorization vector.
- `--defer_dw` deferred weight-gradient accumulation: backward calls stash their (x, dy) rows per Linear and one GEMM per layer per batch flushes them (`model_flush_grads`), via existing kernels only. **Bit-identical** to the immediate path (pinned by a new `test_parallel` check, and the variant phase now runs deferred) — and measured **neutral** on the reference 4-core container, refuting the dW-memory-traffic hypothesis: the backward is compute-bound at these model sizes. Off by default; `bench` reports a third `fwd+bwd deferred` figure.
- `--rotframe`: the cv curvature correction predicted in the motion-aligned (along/cross-track) frame and rotated into lat/lon by the velocity direction — parameter-free, with an exact hand-written backward through the frame Jacobian ∂u/∂v, gradient-checked standalone and through the parallel trainer. Off by default: the five-seed test came back **negative** (worse at 6h, neutral at 48h — FINDINGS §13).
- Text-free data-scaling path: `tools/hurdat2_to_csv.py` converts raw NOAA HURDAT2 databases to the repo's CSV schema (`--since=2004` default, missing values mapped like the bundled sample), and `--emb=none` trains without embedding chunks — the language branch is fed zeros (== `--no_text`, which FINDINGS measured as marginally better than real text), so scaling to 20+ years of storms needs no GPT-4o/MiniLM regeneration.
- **Accuracy features** (from the optimization review): `--physics` (7 second-order features: acceleration, translation speed, heading unit vector, seasonal day-of-year phase; composes with `--motion`, auto-detected at eval by width arithmetic); ensemble evaluation (`eval --weights=a.ckpt,b.ckpt,...` averages predictions in normalized space, config-checked); loss shaping (`--huber=δ` Huber loss on the normalized residual, `--hweight=γ` mean-normalized horizon weights `(h+1)^γ` — both exact-gradient, covered by a new `test_model` FD check); split-restricted scoring (`eval --split=test --split_seed=S` re-derives the training partition and scores only the held-out storms).
- `make MARCH=x86-64-v3`: portable SIMD tier (AVX2+FMA) between plain `-O3` and `NATIVE=1`.
- `eval --threads=N`: evaluation sharded across ParTrainer replicas (also used for the per-epoch validation pass in training) — identical metrics, near-linear scaling.

### Changed
- **The recommended recipe is now `--motion --physics --direct --huber=0.1`** — the direct multi-horizon head replaces `--cv` as the default decoder (FINDINGS §22). It beats the autoregressive `--cv` head at every horizon (32.1 vs 35.2 km @ 6h, level with CLIPER; −10 km over 6–48h, 5/5 seeds), trains ~1.2× faster (no 8-step rollout), and does not need `--swa` (which it subsumes). Recipe guidance updated across README, root README, CLAUDE.md, AGENTS.md, and MEMORY; historical FINDINGS §1–§21 and prior CHANGELOG entries keep their `--cv` numbers (that is the head they were measured on). `--cv` remains fully supported for reference and reproducibility.
- **`--km_loss` reworked into a proper equirectangular km objective** (roadmap Item 4). Was a serial-only gradient hack that scaled only the longitude *gradient* by cos²(lat), leaving the reported loss as plain MSE (inconsistent) and ignoring the coordinate-std anisotropy. Now lives in `model_loss`: scales the longitude *residual* by `(cstd_lon/cstd_lat)·cos(lat)` so loss ≈ squared km error, with the gradient reweighted consistently (chain-rule factor gradient-checked — new `km_loss` / `km_loss+huber` cases in `test_model`), and applied on **both** the serial and data-parallel paths (no longer serial-only). Off by default and bit-identical to MSE when off (golden 0.02507 unchanged). Measured neutral 5-seed (FINDINGS §21) — kept as a correct, documented option.
- **No-spatial encoder is now the default.** The paper's N=1 spatial blocks (Q/K never train; accuracy-neutral per FINDINGS §7) are skipped unless `--spatial` is passed (required to load pre-change checkpoints; `--no_spatial` is accepted as a no-op). Full config drops 5.14M → 2.77M params and ~1.75× encoder compute.
- Unrolled/contiguous restructuring of the remaining reduction loops: `linear_backward`'s bias reduction and all six strided single-accumulator loops in `mha_forward`/`mha_backward`. Full-config bench (portable, with the new default): forward 4.1 ms/sample, forward+backward 14.5 ms — vs 10.8/39.6 before this round and 60.3/84.0 at the start of the optimization work; epoch time ~23 s portable / ~18 s NATIVE. Golden re-pinned (0.02700 → 0.02507) for the intentional architecture-default + summation-order change.
- Data-parallel (`--threads=N`) training support for the previously serial-only paths: `--cv`, `--gru`, `--xattn`, and `--co_spatial`. The workers now feed the same per-sample auxiliary inputs as the serial loop (seed velocity for the cv-anchored decoders, co-active neighbour tables for co-spatial attention) — the replicas were already built with the right architecture, the aux plumbing was the missing piece. `tests/test_parallel` gained a variant phase that pins serial/parallel gradient equivalence for each of the four on the real dataset (~1e-8); end-to-end multicore training reproduces the serial held-out results within dropout-stream noise (cv seed 5: 38.4 vs 37.8 km). `--km_loss` remains serial-only.

### Changed
- `mat_matmul_bt` (the forward-pass workhorse behind every `linear_forward`) now splits its dot products across 8 independent accumulators, breaking the loop-carried float-addition dependency chain so the compiler can pipeline and SLP-vectorize the reduction at plain `-O3`. Full-config forward: 60.3 → 10.8 ms/sample (5.6×); forward+backward: 84.0 → 39.6 ms/sample (2.1×) on the portable build. Summation order changed, so `test_golden`'s pinned loss was updated (0.02706 → 0.02700) per the documented procedure.
- `partrainer_broadcast` copies each parameter tensor with one `memcpy` instead of an element-wise loop (runs over all params × replicas every minibatch).

### Fixed
- **`eval`/`predict` silently scored a 170× wrong number when the decoder variant was not re-specified — the checkpoint now records it (TFW4).** `checkpoint.h` claimed the untagged-parameter size guard "forces EVAL-time flags to match TRAINING-time flags." It does not. `--gru`/`--xattn`/`--posenc` change tensor shapes, so a mismatch trips the guard — but **`--cv`, `--delta` and `--rotframe` are parameter-NEUTRAL**: they reuse the same `fc1`/`fc2` and only change what the decoder's output *means* (a correction to a constant-velocity anchor, versus an absolute coordinate). The byte accounting sails through, the model loads cleanly, and then misreads its own output. Hit for real on the 826-storm run: a cv-trained checkpoint reported **dR 31.02 km** from training and **3726 km** from `eval` on the same weights; the isolated case measured **5071 km vs 29 km**. No warning, no error. The TFW4 header now carries a mode bitfield (`TF_MODE_CV`/`DELTA`/`GRU`/`XATTN`/`ROTFRAME`/`NO_SPATIAL`/`POSENC`/`POOL_LAST`/`CO_SPATIAL`); `eval`/`predict` read it and self-configure, printing e.g. `decoder/arch (from checkpoint): cv+no_spatial`. Older checkpoints (TFW1–3) still load and now print a loud warning that they cannot self-describe. Round-trip pinned in `tests/test_checkpoint`.
- **The CUDA backend could never have run, and now does.** It was only ever *compile*-checked (CI has no GPU), and the first execution on a real device — a Quadro P1000 (sm_61, CUDA 12.8) — **segfaulted instantly**. `tensor_cuda.cu` made `Mat.data` a `cudaMalloc`'d **device** pointer, on the stated assumption that "the rest of the program is oblivious to where the data lives." It is not: `nn.c` and `model.c` dereference `Mat.data[i]` **directly in ~189 places** (weight init, loss, softmax, all the glue), as does the backend cross-check. The first host read of a device pointer is a segfault, so the backend was structurally incompatible with its own callers. `Mat.data` is now **host-resident with per-call device staging** — exactly the choice `tensor_opencl.c` documents and the reason *that* backend works. Verified on hardware: kernel cross-check exact to **1.19e-07** (tol 1e-04), and the tensor + full-model gradient checks pass **through the GPU kernels**.
- **CUDA device-buffer pool.** `cudaMalloc`/`cudaFree` cost ~1 ms *each*; issuing a malloc/free pair per matrix op left the full-model gradient check still grinding after **30 minutes** — the allocator, not the kernels, was the entire cost. Three grow-on-demand device slots (safe because every launch is followed by `cudaDeviceSynchronize()`) removed it from the hot path.
- **CI's `build-test` job had been red on `main` since 2026-07-12 — four merges landed on a failing build.** The job runs `make` → `make test` → `make test-san` → smoke test. `test-san` is `make clean && make SAN=1 test`, so it leaves **ASan/UBSan-instrumented objects** behind; the smoke step's plain `make` then saw those `.o` files as up to date, merely **relinked** them without `-fsanitize`, and died on undefined `__asan_*` / `__ubsan_*` symbols. Reproduced locally, then fixed by cleaning before the smoke build. (The gcc/clang matrix was the only red job — `valgrind`, `opencl` and `cuda-compile` all stayed green, which is why it went unnoticed.)
- **The RNG was silently broken on Windows: every weight initialized to the bottom of its range.** The xorshift64 state (weight init, dropout, and all three storm-split shuffles) was held in `unsigned long` — 64-bit on LP64 (Linux/macOS) but **32-bit on LLP64 (Windows)**. At 32 bits the seed truncates (88172645463325252 → 3418323524), `(13,7,17)` is not a full-period triple, and — fatally — `nn_uniform` takes `xorshift() >> 11` and divides by `2^53`, so with only ~21 significant bits left **every draw collapses to ~0**. Measured on Windows: `nn_uniform(-0.5, 0.5)` returned `-0.499999999986, -0.499999999935, -0.499999999931, …` — every parameter starts at the low end, the network is born perfectly symmetric, and `test_nn`'s finite-difference gradient check fails (72 params, max rel err 9.99e-01). It compiled and ran; the model was **silently garbage**. The state is now `uint64_t` throughout (`nn.c`, `data.c`, `parallel.c`, `train.c`, and the `nn.h`/`data.h` signatures), plus the `0x100000001b3` FNV multiplier in `parallel.c` which was truncating too. **This is a no-op on LP64** — `unsigned long` *is* 64-bit there — so the golden value is unchanged and CI is unaffected; `test_golden` (which pins the training loss bit-exactly) now **passes on Windows**, proving the two platforms generate an identical stream.
- `dataset_split` clamps the validation count to `[0, N]`: an out-of-range `val_frac` made `N - nv` negative, which wraps to a huge `size_t` in the following `malloc` (gcc's `-Walloc-size-larger-than=` flags exactly this).
- `tests/test_parallel` was passing vacuously: its synthetic dataset left the coordinate std at 0 (division by zero → NaN losses), and the NaN-blind `>` threshold comparison reported "ok" regardless. Fixed the dataset stats and made the comparison NaN-proof; the test now genuinely proves serial/parallel gradient equivalence (max abs diff ~1e-8).

## 2026-07-11 — the pure-C reimplementation

The bulk of the current codebase: `typhoformer-c/`, a clean-room, dependency-free C11 port of the full pipeline, plus the honest re-evaluation of the model it enabled.

### Added
- **C port core**: tensor primitives with cache-blocked matmuls (`tensor.c`); NN layers with hand-written forward *and* backward (Linear, LayerNorm, FFN, multi-head attention, transformer block — `nn.c`); the full model (PGF fusion, spatio-temporal encoder, autoregressive decoder — `model.c`); CSV/`.npy` data loader with sliding windows (`data.c`); Adam optimizer and end-to-end training loop. Every layer validated by finite-difference gradient checks.
- **CLI**: six subcommands — `train`, `eval`, `prepare`, `predict`, `baseline`, `bench` — with full hyperparameter flags, multi-step autoregressive decoding, and per-horizon metrics (MAE + spherical ΔR).
- **Training hygiene**: dropout, AdamW (decoupled weight decay), gradient clipping, LR warmup/decay, early stopping, best-checkpoint saving, resume with optimizer sidecar.
- **Extension seams**: pluggable `Module` vtable + `Sequential` (`module.h`); a documented ~13-kernel compute-backend seam (`backend.h`) with a runnable **OpenCL** backend (verified end-to-end via POCL) and a **CUDA** backend (nvcc-clean); data-parallel multicore training via pthreads (`--threads=N`), gradient-equivalence-tested against serial.
- **Experiment flags**: `--motion` (position + velocity inputs) and `--delta` (displacement head); `--no_text` language-branch ablation and `--split_seed`; encoder options (`--no_spatial`, `--posenc`, `--pool=last`, `--prenorm`, `--timebias`, `--co_spatial`); decoder variants `--cv` (constant-velocity anchor), `--gru` (recurrent rollout state), `--xattn` (cross-attention over the encoder sequence); `--km_loss` (tested, did not help — off by default).
- **Tests + CI**: gradient checks (tensor/nn/model), golden-loss regression, module, parallel-equivalence, checkpoint round-trip, `.npy` validation, dropout, GRU, cross-attention; gcc+clang CI matrix with sanitizers, valgrind, an OpenCL job (POCL), a CUDA compile-check, and a subcommand smoke test.
- **Offline tools** (`tools/`): GPT-4o description generation, MiniLM embedding generation, `.npy`-dict → `.tfb` converter.
